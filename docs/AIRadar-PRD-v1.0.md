## 2. 产品概述

### 2.1 产品背景
全球AI行业正处于高速发展阶段，每日产生海量资讯——涵盖模型发布、学术突破、商业融资、政策法规等多个维度。然而，优质信息高度分散，且绝大多数来自英文媒体，国内AI从业者面临以下痛点：
- 信息分散：需同时关注数十个英文来源，成本极高
- 语言障碍：技术类英文阅读门槛高，翻译工具质量参差不齐
- 噪音过滤难：无法有效区分「重大进展」与「营销软文」
- 时效滞后：国内媒体转载周期通常滞后24-72小时

### 2.2 产品定位
**产品核心主张**：AIRadar 是一个面向AI从业者的「全球AI资讯智能聚合平台」，每日自动抓取全球30+顶级AI信息源，经AI流水线处理后，以中英双语、分类标签、重要性评分的形式呈现，帮助用户在5分钟内掌握全球AI动态。

### 2.3 目标用户
| 用户类型 | 规模估算 | 核心诉求 | 使用场景 |
|---|---:|---|---|
| AI创业者 / 产品经理 | 约200万人 | 竞品动态、技术趋势、融资情报 | 每日早晨通勤、工作前 |
| AI研究员 / 工程师 | 约100万人 | 最新论文、模型发布、开源动态 | 碎片时间、午休 |
| 投资人 / 分析师 | 约20万人 | 融资并购、市场动向、政策监管 | 早报、周报汇总 |
| 科技媒体从业者 | 约10万人 | 一手信息源、选题参考 | 日常工作 |
| AI学习者 / 爱好者 | 约500万人 | 了解行业动态、学习资料 | 睡前阅读 |

### 2.4 产品目标（OKR）
**Objective 1：打造行业最权威的AI资讯聚合入口**
- KR1：上线6个月内日活用户（DAU）突破5万
- KR2：信息源覆盖量达到30+，内容更新频率≥3次/天
- KR3：用户平均会话时长≥8分钟

**Objective 2：建立可持续的商业变现能力**
- KR1：付费会员转化率≥3%
- KR2：上线12个月内月收入突破50万元
- KR3：企业客户签约≥20家

- RSS优先：优先使用官方RSS/Atom Feed，稳定性高，无反爬风险

- 人工补录：对重大事件可由运营人员手动录入，绕过自动化限制

### 4.2 爬取与数据处理模块
#### 4.2.1 定时爬取任务（UTC+8）
- 早间抓取：06:00（覆盖美国西海岸当日新闻）
- 午间抓取：12:00（覆盖欧洲上午及亚洲早间发布）
- 晚间抓取：18:00（覆盖美国东部上午新闻）
- 紧急推送：监测到大型事件（如GPT-5发布）触发即时爬取

#### 4.2.2 内容去重机制
- URL指纹：对原文URL进行MD5哈希，入库前校验唯一性
- 语义去重：使用文章标题Embedding向量，余弦相似度 > 0.85 视为重复，保留权重较高来源

#### 4.2.3 重要性评分算法（0-100）
| 维度 | 权重 | 说明 |
|---|---:|---|
| 来源权重 | 30% | OpenAI官博=10分，TechCrunch=8分，其余按媒体影响力分级 |
| 关键词密度 | 25% | 命中核心关键词（GPT/Claude/Gemini等）加分 |
| 社交热度 | 25% | HackerNews积分 + Reddit upvotes（API获取） |
| 时效性 | 20% | 24小时内满分，每过6小时衰减10% |

### 4.3 AI内容处理流水线
#### 4.3.1 处理流程
语言检测 → 全文提取 → 重要性评分 → 中文摘要生成 → 分类标签 → 关键实体识别 → 入库

#### 4.3.2 中文摘要规格
- 字数：200-300字中文摘要，覆盖 Who/What/Why/Impact 四要素
- 核心要点：自动提炼3条 Bullet Point（每条≤30字）
- 难词解释：对术语（如RAG、RLHF）自动附注中文解释
- 情感判断：标注「利好/利空/中性」（投资者视角）

#### 4.3.3 分类标签体系
| 一级分类 | 二级分类示例 |
|---|---|
| 大模型 | 预训练、微调、推理、多模态、开源模型 |
| AI应用 | AI编程、AI写作、AI图像、AI视频、AI医疗 |
| 芯片与算力 | GPU、TPU、AI芯片、边缘计算、算力市场 |
| 政策与监管 | 美国、欧盟、中国、数据法规、AI安全 |
| 融资与并购 | VC融资、A轮/B轮、IPO、战略收购 |
| 学术研究 | 论文速递、基准测试、新技术方向 |
| 产品发布 | 新功能、API更新、定价变化、开源发布 |

### 4.4 前端功能需求
#### 4.4.1 首页
- 今日精选：当日重要性评分Top 10文章
- 资讯流：按时间倒序+评分混合排序
- 分类快筛：顶部Tab按一级分类切换
- 热门标签云：展示当日高频关键词
- 数据看板：今日抓取条数、涵盖媒体数量

#### 4.4.2 文章列表卡片
- 来源媒体Logo + 媒体名称
- 发布时间（如「3小时前」）
- 文章标题（中英双语）
- 三行摘要预览
- 分类标签（最多3个）
- 重要性评分徽标（🔥热门 / ⭐精选 / 🎓学术）

#### 4.4.3 详情页
- 中文摘要全文（含核心要点、情感判断）
- 原文跳转按钮（带UTM追踪参数）
- 相关文章推荐（基于分类+关键词）
- 分享按钮（复制链接 / 微信 / 微博）
- 用户反馈：摘要是否准确（👍👎）

#### 4.4.4 搜索与筛选
- 全文搜索：支持中英文关键词（标题+摘要）
- 筛选维度：分类 × 来源 × 时间范围 × 重要性
- 搜索历史：本地存储最近10次搜索记录

### 4.5 订阅系统
#### 4.5.1 邮件早报
- 订阅方式：输入邮箱即可订阅，无需注册
- 推送时间：每日07:30（UTC+8）
- 内容结构：当日精选5条 + 分类亮点各1条 + 明日预告
- 个性化：付费用户可自定义关注分类和关键词

#### 4.5.2 RSS输出
- 提供全站RSS Feed（`/rss.xml`）
- 提供分类RSS Feed（`/rss/{category}.xml`）
- 支持用户输出自定义关键词RSS（付费功能）

## 5. 非功能需求

### 5.1 性能要求
| 指标 | 要求 | 备注 |
|---|---|---|
| 首页加载时间 | ≤ 1.5秒（LCP） | CDN加速，静态化处理 |
| 搜索响应时间 | ≤ 500ms | Elasticsearch全文索引 |
| 爬取成功率 | ≥ 95% | 失败自动重试3次 |
| AI处理延迟 | ≤ 30秒/篇 | 批量处理，并行调用 |
| 系统可用性 | ≥ 99.5% | 月均宕机时间≤3.6小时 |
| 并发支持 | 1000 QPS | 初期目标，水平扩展架构 |

### 5.2 安全要求
- 反爬合规：遵守robots.txt，设置合理爬取间隔，不绕过付费墙
- 数据安全：用户邮箱等PII数据加密存储，不对外泄露
- XSS/CSRF防护：前端输入做转义，API接口加Token验证
- 速率限制：单IP每分钟最多60次请求

### 5.3 合规要求
- 版权：只存储摘要，不全文转载；详情页标注原文来源链接
- AI标识：AI生成摘要明确标注「AI摘要」
- 隐私政策：邮箱订阅需明确说明数据用途

## 6. 技术架构概览

### 6.1 技术选型
| 层次 | 技术选型 | 选型理由 |
|---|---|---|
| 前端 | Next.js 14 + TypeScript + Tailwind CSS | SSR支持SEO，开发效率高 |
| 后端API | Python FastAPI | 异步高性能，AI生态完善 |
| 爬虫调度 | Celery + Redis（Broker） | 分布式任务队列，支持定时任务 |
| 爬虫引擎 | Scrapy + Playwright | 静态页RSS优先，JS页面用Playwright |
| 数据库 | PostgreSQL（主库）+ Redis（缓存） | 关系型存储文章，Redis缓存热点 |
| 搜索引擎 | Elasticsearch | 全文搜索，支持中文分词 |
| AI处理 | Claude API（claude-sonnet-4） | 摘要质量高，支持中文输出 |
| 文件存储 | AWS S3 / 阿里云OSS | 存储原文快照，防止来源失效 |
| 部署 | Docker + Kubernetes + Nginx | 容器化部署，水平扩展 |
| 监控 | Prometheus + Grafana + Sentry | 性能监控+异常告警 |

### 6.2 数据流架构
信息源（RSS/网页） → 爬虫调度器（Celery） → 内容提取 → 去重检查 → AI处理流水线（Claude API） → PostgreSQL + Elasticsearch → 前端API → 用户展示 / 邮件推送
